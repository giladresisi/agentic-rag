# Fix Retrieval Similarity Threshold

**Complexity**: ✅ Simple

## Problem
- Document ingestion works correctly
- Retrieval tool is properly implemented and called by LLM
- **BUT** similarity threshold (0.7) is too high for real-world use
- Best similarity score observed: 0.417
- Result: Tool returns empty results → LLM says "I'm unable to access..."

## Solution
1. Lower default similarity threshold from 0.7 to 0.4
2. Add optional system prompt to encourage tool use
3. Create validation test

## Tasks

### Task 1: Lower similarity threshold in config
**File**: `backend/config.py`
**Change**: Line 45
```python
# From:
RETRIEVAL_SIMILARITY_THRESHOLD: float = 0.7

# To:
RETRIEVAL_SIMILARITY_THRESHOLD: float = 0.4
```

**Validation**: Run test to confirm retrieval works with new threshold

### Task 2: Add system prompt support to chat service
**File**: `backend/services/chat_service.py`
**Add**: System message if conversation doesn't start with one

Add after line 78 (in stream_response method):
```python
# Add system prompt if not present
if not conversation_history or conversation_history[0].get("role") != "system":
    system_message = {
        "role": "system",
        "content": "You are a helpful assistant with access to the user's uploaded documents. When users ask about their documents or specific information, use the retrieve_documents tool to search their knowledge base. Always check the uploaded documents before saying you don't have access to information."
    }
    conversation_history = [system_message] + conversation_history
```

**Validation**: Verify system message is included in LLM requests

## Expected Outcome
- Retrieval returns results for relevant queries
- LLM receives document content and answers questions
- User query "What's written in the 1st line?" gets answered correctly

## Trade-offs
- Lower threshold (0.4) may return less relevant results in some cases
- But 0.7 is too strict for real-world use
- Can be tuned per-query if needed via API parameter
