# Feature: Enhanced Settings with OpenAI, OpenRouter, and LM Studio Provider Support

Validate documentation and codebase patterns before implementing. Pay special attention to existing provider configurations, API key handling, and database constraints.

## Feature Description

Enhance the settings modal to support provider-specific configuration for both Chat (LLM calls) and Documents (embeddings):
- **Limit providers to OpenAI, OpenRouter, and LM Studio** (remove Ollama, Custom)
- **OpenAI**: Show dropdown of available models for chat and embeddings
- **OpenRouter**: Show text inputs for model name and dimensions (embeddings only)
- **LM Studio**: Show text inputs for model name and base URL (for local models)
- **Separate API keys**: Use OPENAI_API_KEY, OPENROUTER_API_KEY, LM_STUDIO_API_KEY
- **Dynamic embedding dimensions**: Support models with different vector dimensions (requires database migration)
- **Embedding model lock**: Prevent changing embeddings config when chunks exist in database

## User Story

As a user
I want to configure OpenAI, OpenRouter, or LM Studio as my provider for chat and embeddings separately
So that I can use cloud or local models for conversations vs document processing, and prevent accidental embedding changes when documents exist

## Problem Statement

Current issues:
- Settings modal shows 5 providers but user wants OpenAI, OpenRouter, and LM Studio only
- No support for OpenRouter or LM Studio API keys (uses OPENAI_API_KEY for all providers)
- Embedding dimensions hardcoded to 1536 in database (blocks text-embedding-3-large)
- No provider-specific UI (same fields for all providers)
- Embedding service hardcoded to OpenAI only
- No way to use OpenRouter or LM Studio for embeddings
- Users can change embedding models even when documents exist, causing dimension mismatches

## Solution Statement

Implement provider-specific configuration UI:
- Limit provider selection to OpenAI, OpenRouter, and LM Studio
- Show OpenAI model dropdowns (chat: 4 models, embeddings: 3 models)
- Show OpenRouter text inputs for model name and dimensions
- Show LM Studio text inputs for model name and base URL
- Add OPENROUTER_API_KEY and LM_STUDIO_API_KEY environment variables
- Update backend to route API keys based on provider
- Enhance embedding service to support multiple providers
- Provide database migration path for variable embedding dimensions
- Lock embeddings configuration when chunks exist (prevent accidental changes)

## Feature Metadata

**Type**: Enhancement | **Complexity**: High | **Breaking**: Yes (DB schema change)
**Affected**: Frontend (Settings UI, types), Backend (config, provider service, embedding service), Database (vector dimensions)

## NOTES

### Design Decisions

1. **Provider Limitation**: Support OpenAI, OpenRouter, and LM Studio only (removed Ollama and Custom)
2. **Clean Architecture**: provider_service handles all provider-specific logic; chat/embedding services are provider-agnostic
2. **Provider Abstraction**: provider_service acts as adapter layer - chat/embedding services are provider-agnostic
3. **Dimension Handling**: Support variable dimensions with migration path
4. **Base URL**: Auto-populated for OpenAI/OpenRouter, always user-provided for LM Studio (no default)
5. **Embedding Lock**: Prevent changes when chunks exist to avoid dimension mismatches
6. **Backward Compatibility**: Migration adds columns with default values (1536) to maintain existing data

### Migration Strategy

When user changes embedding dimensions:
1. Update embedding config in settings (new model + dimensions)
2. Upload new documents - they get embedded with new dimensions
3. Old documents remain with old dimensions
4. Run re_embed_documents.py script to migrate existing documents
5. Script deletes old chunks and creates new ones with new dimensions
6. Retrieval function filters by dimension to avoid mixing

### Trade-offs

**Pros:**
- Clean separation of chat vs embedding configuration
- Provider-specific UI (dropdown vs text input)
- Support for multiple embedding dimensions
- Backward compatible (existing data not broken)

**Cons:**
- Re-embedding is manual process (not automatic on config change)
- Mixed dimension data possible during migration
- Database migration required for dimension changes
- Cannot mix dimensions in single retrieval (by design)

### Future Enhancements

- Auto-detect when dimension change occurs and prompt user to re-embed
- Background job to re-embed documents automatically
- Migration progress indicator in UI
- Dimension validation before upload (warn if mismatch)
- Support for more providers (Anthropic, Cohere, etc.)
