# Feature: Migrate from OpenAI Assistants API to Responses API

## Context

The OpenAI Assistants API was deprecated on August 26, 2025, with a sunset date of August 26, 2026. OpenAI has introduced the **Responses API** as the official replacement, offering improved performance, simpler architecture, and better integration with agentic workflows.

**Current State:**
- Using Assistants API with managed threads (`client.beta.threads.*`)
- Requires pre-configured Assistant ID via OpenAI dashboard
- Database stores `openai_thread_id` for thread management
- Streaming uses `thread.message.delta` events
- File: `backend/services/openai_service.py`

**Target State:**
- Use Responses API with manual history management
- No Assistant objects needed
- Stateless approach - store and manage conversation history in database
- Streaming uses `response.output_text.delta` events
- Align with project philosophy: "stateless completions - store and send chat history yourself" (CLAUDE.md)

**Why This Migration:**
1. Assistants API is deprecated (sunset Aug 26, 2026)
2. Responses API offers 3% performance improvement on reasoning tasks
3. Simpler architecture - single API call per turn vs managing runs/threads
4. Better caching - 40-80% improvement in cache utilization
5. Aligns with project's stateless philosophy

## Problem Statement

The current implementation relies on OpenAI's deprecated Assistants API, which manages threads and conversation state externally. This creates:
- Technical debt (API will be sunset in 2026)
- Dependency on external state management (OpenAI stores conversation history)
- Complexity in thread management
- Requirement for manual Assistant configuration in OpenAI dashboard

## Solution Statement

Migrate to the Responses API using **manual history management** (stateless approach):
1. Remove dependency on OpenAI threads - database becomes source of truth
2. Pass full conversation history in each API call
3. Eliminate Assistant ID requirement
4. Update streaming to use new event format
5. Simplify architecture by removing thread creation/management

This approach aligns with the project's Module 2 vision of stateless completions while providing immediate benefits of the modern Responses API.

    Args:
        messages: List of message dicts with 'role' and 'content' keys

**ADD:** New streaming method using Responses API

**IMPLEMENT:**
```python
@staticmethod
async def stream_response(
    conversation_history: List[Dict[str, str]],
    model: str = "gpt-4o-mini"
) -> AsyncGenerator[str, None]:
    """Stream a response using OpenAI Responses API.

    Args:
        conversation_history: Full conversation history (list of messages)
        model: Model to use for generation

    Yields:
        Text deltas from the streaming response
    """
    try:
        async with async_client.responses.stream(
            model=model,
            input=conversation_history,
            store=False,  # Stateless - no data retention
        ) as stream:
            async for event in stream:
                if event.type == "response.output_text.delta":
                    yield event.delta
                elif event.type == "error":
                    raise Exception(f"OpenAI API error: {event}")
    except Exception as e:
        raise Exception(f"Failed to stream response: {str(e)}")
```

**PATTERN:** Follow async generator pattern from existing `stream_message` method

**IMPORTS:**
```python
from typing import List, Dict, AsyncGenerator
```

**GOTCHA:**
- Must use `async_client.responses.stream()` not `client.responses.stream()`
- Event type is `response.output_text.delta` not `thread.message.delta`
- Set `store=False` for stateless, zero data retention

## NOTES

### Design Decision: Manual History Management

**Chosen Approach:** Manual history management (stateless)
- Send full conversation history in each API call
- Set `store=False` for zero data retention
- Database is source of truth for conversation history

**Rationale:**
1. Aligns with CLAUDE.md philosophy: "stateless completions - store and send chat history yourself"
2. Simpler than Conversations API (no conversation_id to manage)
3. Full control over history and data retention
4. Prepares codebase for Module 2 (provider flexibility)

**Alternative Considered:** Conversations API
- Would require storing `conversation_id` in database
- Automatic history management by OpenAI
- Less control, not truly stateless
- Rejected: Doesn't align with project goals

### File Search Feature

**Decision:** Deferred to Module 2

The current implementation uses file_search via the Assistant. To migrate this:
1. Would need to create Vector Stores in OpenAI
2. Upload files to Vector Stores
3. Pass `vector_store_ids` in tools parameter

**Recommendation:** Remove file_search from Module 1, re-implement in Module 2 with full RAG pipeline (chunking, embeddings, pgvector). This is cleaner than half-migrating now.

If user wants to keep file_search in Module 1:
- Add Vector Store creation to setup instructions
- Update `stream_response()` to accept tools parameter
- Pass `tools=[{"type": "file_search", "vector_store_ids": [...]}]`

### Performance Expectations

Based on OpenAI benchmarks:
- **3% improvement** in reasoning tasks (SWE-bench)
- **40-80% better cache utilization** vs Chat Completions
- **Lower costs** due to improved caching
- **Faster responses** due to simplified architecture

### Breaking Changes Summary

What's removed:
- OpenAI thread management
- Assistant ID requirement
- Manual Assistant configuration in dashboard
- `openai_thread_id` database field

What's preserved:
- Frontend code (no changes)
- SSE streaming format (backward compatible)
- Message persistence in database
- Thread isolation and RLS
- Authentication flow
