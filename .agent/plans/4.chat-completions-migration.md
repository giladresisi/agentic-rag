# 4. Chat Completions API Migration + Provider/Model UI Selector

**Complexity:** ⚠️ **Medium** - Requires both backend API changes and new frontend UI

**Parallel Execution:** ✅ **Yes** - Can run in parallel with Plan 5 (Document Ingestion)

## Parallel Execution Strategy

This plan can be executed by a team of 3 agents working in parallel:

### Team Structure
- **Agent 1 (Backend-API)**: Provider service, chat router updates, config changes
- **Agent 2 (Backend-Core)**: OpenAI service refactor (Responses API → Chat Completions)
- **Agent 3 (Frontend)**: Provider selector UI, chat interface integration, hooks

### Task Dependencies
- Agent 1 and Agent 2 can work in parallel (minimal overlap)
- Agent 3 can start after Agent 1 completes provider service (needs `/providers` endpoint)
- Final integration: Agent 2's streaming changes + Agent 3's UI → full flow testing

### Execution Order
1. **Phase 1 (Parallel):**
   - Agent 1: Build provider service and `/providers` endpoint
   - Agent 2: Refactor OpenAI service to Chat Completions API

2. **Phase 2 (After Agent 1):**
   - Agent 3: Build frontend provider selector and integration

3. **Phase 3 (Integration):**
   - All agents: Integration testing and validation

## Context

Module 1 uses OpenAI's Responses API which manages threads, memory, and file search internally. This phase migrates to the standard Chat Completions API to:
- Support any OpenAI-compatible provider (OpenAI, OpenRouter, Ollama, LM Studio, etc.)
- Build provider-agnostic LLM clients
- Give users control over which model/provider they use

**Architectural Decision:** Replace Responses API entirely (Option A from PRD). The existing stateless pattern (fetching conversation history from DB) is already compatible with Chat Completions API.

## What This Phase Delivers

- Generic OpenAI-compatible Chat Completions client
- Frontend UI for selecting provider and model per conversation
- Provider presets (OpenAI, OpenRouter, Ollama, LM Studio, Custom)
- Model validation and error handling
- All existing chat functionality preserved (streaming, multi-turn, RLS)
- LangSmith observability maintained

## Backend Changes

### 1. Configuration (`backend/config.py`)
**Changes:**
- Remove: `OPENAI_VECTOR_STORE_ID` (no longer using Responses API file_search)
- Add: `DEFAULT_PROVIDER`, `DEFAULT_MODEL`, `DEFAULT_BASE_URL` (fallback values)

### 2. OpenAI Service (`backend/services/openai_service.py`)
**Changes:**
- Replace `async_client.responses.stream()` with `async_client.chat.completions.create(stream=True)`
- Update streaming event parsing for Chat Completions format
- Accept provider configuration parameters (base_url, model, api_key)
- Update LangSmith tracing to log Chat Completions calls

**Event parsing change:**
```python
# OLD (Responses API)
if event.type == "response.output_text.delta":
    yield event.delta

# NEW (Chat Completions API)
if chunk.choices[0].delta.content:
    yield chunk.choices[0].delta.content
```

### 3. Chat Router (`backend/routers/chat.py`)
**Changes:**
- Update `send_message` endpoint to accept provider configuration
- Pass provider config to `openai_service.stream_response()`

**Request body update:**
```python
class SendMessageRequest(BaseModel):
    content: str
    provider: str = "openai"
    model: str = "gpt-4o-mini"
    base_url: str | None = None
    api_key: str | None = None  # Optional override
```

### 4. Provider Service (`backend/services/provider_service.py` - NEW)
**Purpose:** Central configuration for provider presets

**Provider presets:**
- OpenAI: `https://api.openai.com/v1` with models: gpt-4o, gpt-4o-mini, etc.
- OpenRouter: `https://openrouter.ai/api/v1`
- Ollama: `http://localhost:11434/v1` (no API key required)
- LM Studio: `http://localhost:1234/v1` (no API key required)
- Custom: user-provided base_url and model

**New endpoint:**
```python
@router.get("/providers")
async def get_providers():
    return {"providers": PROVIDER_PRESETS}
```
